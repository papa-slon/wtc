## Findings
1. Severity: High. The existing `/admin/system-health` LMS card is the correct minimal home for the acknowledgement/retry workflow. Evidence: the page already admin-gates itself at `apps/web/src/app/admin/system-health/page.tsx:8`-`apps/web/src/app/admin/system-health/page.tsx:13`, derives LMS cleanup status from aggregate counts at `apps/web/src/app/admin/system-health/page.tsx:19`-`apps/web/src/app/admin/system-health/page.tsx:20`, and renders the "LMS upload cleanup review" card at `apps/web/src/app/admin/system-health/page.tsx:74`-`apps/web/src/app/admin/system-health/page.tsx:118`. Recommendation: extend this card with compact operator controls; do not add a new admin route, nav item, or row table. Target part: `apps/web/src/app/admin/system-health/page.tsx`.
2. Severity: High. The current frontend DTO is intentionally aggregate-only, so the UI cannot safely address individual cleanup tasks. Evidence: `LmsObjectCleanupOpsSummary` exposes provider, reason, counts, timestamps, and a generic error code only at `apps/web/src/features/admin/types.ts:54`-`apps/web/src/features/admin/types.ts:66`; `loadSystemHealth()` maps only those fields at `apps/web/src/features/admin/queries.ts:199`-`apps/web/src/features/admin/queries.ts:211`; the DB summary selects status/reason/attempts/runAfter/error/timestamps, not IDs or keys, at `packages/db/src/repositories.ts:982`-`packages/db/src/repositories.ts:998`; the UI/static test forbids `storageKey`, `cleanupTaskId`, filename, hash, auth, and `X-Amz` strings at `tests/integration/admin-lms-cleanup-review.test.ts:19`-`tests/integration/admin-lms-cleanup-review.test.ts:22`. Recommendation: frontend forms must submit no task IDs, object keys, filenames, hashes, signed URLs, or provider data; any action must operate on a server-selected cohort such as "current dead-letter count observed at latestDeadLetteredAt". Target part: admin DTO/action boundary.
3. Severity: High. The backend state machine is not yet ready for a visible retry button unless an aggregate admin contract is added first. Evidence: `lms_object_cleanup_tasks` currently has only `pending`, `completed`, and `dead_letter` states at `packages/db/src/schema.ts:302`-`packages/db/src/schema.ts:324`; the existing completion helper accepts internal IDs and updates only pending rows at `packages/db/src/repositories.ts:869`-`packages/db/src/repositories.ts:877`; the worker candidate reader exposes `cleanupTaskId` and `storageKey` for worker DELETE only at `packages/db/src/repositories.ts:942`-`packages/db/src/repositories.ts:979`. Recommendation: do not wire UI buttons directly to existing worker/internal helpers. Add dedicated aggregate admin operations first: acknowledge current dead-letter cohort, and requeue a bounded dead-letter cohort for worker processing; both should reselect rows server-side and audit summary-only counts. Target part: `@wtc/db` repository contract plus admin actions.
4. Severity: Medium. Existing admin mutation UX already has the right form pattern. Evidence: admin actions document the required chain `requireUser -> assertAdmin -> assertCsrf -> Zod -> repo -> revalidatePath` at `apps/web/src/features/admin/actions.ts:1`-`apps/web/src/features/admin/actions.ts:18`; `CsrfField` is the hidden field for mutating forms at `apps/web/src/lib/csrf.tsx:24`-`apps/web/src/lib/csrf.tsx:34`; the billing manual-review page uses compact card-local forms with `CsrfField`, hidden fields, notes, and buttons at `apps/web/src/app/admin/entitlements/review/page.tsx:120`-`apps/web/src/app/admin/entitlements/review/page.tsx:178`. Recommendation: use two small card-local forms: "Acknowledge incident" with a required note, and "Requeue for worker" with a required note or fixed reason; include only CSRF, operation name, expected counts, and observed latest timestamp. Target part: admin action forms and schemas.
5. Severity: Medium. Operator copy should remain compact and explicitly no-leak. Evidence: current card copy states the private table is count-only and hides cleanup task IDs, object keys, filenames, hashes, signed URLs, scanner details, and provider bodies at `apps/web/src/app/admin/system-health/page.tsx:103`-`apps/web/src/app/admin/system-health/page.tsx:111`; the DTO exposes only a generic `latestDeadLetterErrorCode` at `apps/web/src/features/admin/types.ts:63`-`apps/web/src/features/admin/types.ts:66`; tests assert the no-locator copy and forbidden fields at `tests/integration/admin-lms-cleanup-review.test.ts:14`-`tests/integration/admin-lms-cleanup-review.test.ts:22`. Recommendation: keep labels short: "Acknowledge incident" and "Requeue for worker"; use helper text such as "Applies to the current count only. The worker performs object deletes." Target part: `/admin/system-health` copy.
6. Severity: Low. The workflow should stay card-local and responsive, not become a queue table. Evidence: the current LMS cleanup card uses wrapped metric rows at `apps/web/src/app/admin/system-health/page.tsx:86`-`apps/web/src/app/admin/system-health/page.tsx:101`; shared UI primitives already support card headers, status pills, metric cards, and button classes at `packages/ui/src/components.tsx:9`-`packages/ui/src/components.tsx:53`. Recommendation: add one compact action row under the current metrics that wraps on mobile; avoid per-row cards, row checkboxes, expandable raw details, or a wide table. Target part: responsive admin UI.
## Decisions
- Keep the Phase 3.36 UI inside the existing `/admin/system-health` LMS upload cleanup review card.
- Use aggregate cohort actions only; no row-level task list, task ID, object key, signed URL, filename, hash, scanner detail, or provider response should enter the UI.
- The frontend should not perform object-store deletes. "Requeue for worker" should only change retry state through a backend aggregate operation, then the worker owns deletion and health/audit reporting.
- Acknowledgement should require a short operator note and should be persisted through a backend-audited summary-only contract before the UI claims an incident is acknowledged.
## Risks
- If the server action only receives "ack all" without an expected count/latest timestamp guard, newly arrived dead letters could be acknowledged without being seen by the operator.
- If acknowledgement is audit-only with no queryable state, the UI can refresh counts but cannot honestly show "acknowledged" versus "not acknowledged"; a small summary state or audit-derived latest acknowledgement view is needed.
- If retry-from-dead-letter is wired to existing internal helpers, it risks leaking task IDs/keys or bypassing the worker-owned object-delete boundary.
