### Finding 2 — HIGH — Spec says `jti uuid PRIMARY KEY`; recommend TEXT not UUID for Drizzle compatibility

**Evidence:** `docs/AXIOMA_HANDOFF_TOKEN_SPEC.md` §Replay Prevention DDL line: `jti uuid PRIMARY
KEY`. The existing schema uses `uuid('id').defaultRandom().primaryKey()` for surrogate keys. For
the jti column the value is **caller-supplied** (from `crypto.randomUUID()` at issuance time), not
DB-generated. Drizzle's `uuid()` type maps to `uuid` in Postgres but has no `.notNull()` issue —
it just requires the caller to supply the value at insert time, with no `.defaultRandom()`.

**Decision:** Use `uuid('jti').primaryKey()` (no `.defaultRandom()`) for the PK. This is correct
semantically: jti is a UUID v4 value generated by the application (not the DB) and treated as an
opaque unique identity token. The type should be `uuid` in Postgres (not `text`) to get implicit
format validation and efficient btree indexing.

Similarly, `sub` should be `uuid('sub').notNull()` (no FK — see Finding 3).

**Recommendation:** DDL as follows in schema.ts:
```
axioma_handoff_jti_revocations.jti: uuid('jti').primaryKey()
axioma_handoff_jti_revocations.sub: uuid('sub').notNull()
```

**Target part:** schema.ts Axioma section, migration 0004

### Finding 3 — HIGH — FK decision: no FK on sub→users(id); nullable no-FK is the correct audit pattern

**Evidence:** `packages/db/src/schema.ts` lines 213-237 (`audit_logs` table), lines 663-693
(`billing_manual_review_items` table). Both `audit_logs.actor_user_id` and
`billing_manual_review_items.user_id` are nullable UUIDs **without** a FK to `users.id` to avoid
cascade deletion destroying the audit/history record. The pattern is consistent: "no FK where
history must survive user deletion."

For `axioma_handoff_jti_revocations.sub`, the spec says: "User account deleted → All pending JTIs
for the user are marked `revoked_at = now()`" (revocation-before-delete behaviour). A cascade
FK would DELETE the rows on user delete, destroying the audit trail. A no-FK nullable
approach would require the sub column to be nullable, losing NOT NULL guarantees.

**Decision (recommended):** Use `sub uuid NOT NULL` with **no FK constraint** on `sub→users(id)`.
Rationale: (a) the jti store is an append-only audit/replay table — rows must survive the user; (b)
the revocation-on-delete behaviour is explicitly specified and requires an app-level sweep, not a
DB cascade; (c) `audit_logs` and `billing_manual_review_items` both use the same no-FK UUID
pattern; (d) NULL sub would weaken integrity guarantees without benefit.

Add a note in the schema comment: `-- sub is a WTC user uuid (no FK: rows survive user deletion; app must sweep on delete)`

**Target part:** schema.ts Axioma section, migration 0004

### Finding 4 — HIGH — Index design: two recommended non-unique indexes beyond the PK

**Evidence:** Spec §Cleanup: `DELETE WHERE expires_at < now() - 1 hour` (worker purge). Spec §Revocation
Behavior: "User account deleted → All pending JTIs for the user are marked revoked_at." Spec option A
consume endpoint uses `WHERE jti = ?` (PK lookup — already covered). The `purgeExpiredHandoffJtis`
worker function needs to scan by `expires_at`. The `revokeHandoffJtiByUser` function (for user-delete
sweep) needs to scan by `sub`.

**Decision (recommended indexes):**
1. `index('ahjr_expires_at_idx').on(t.expiresAt)` — supports the purge worker (`WHERE expires_at < ?`)
2. `index('ahjr_sub_idx').on(t.sub)` — supports per-user revoke sweep on account deletion

No unique index beyond the PK is needed. No composite index needed for the consume path (`WHERE
jti = ? AND ...` hits the PK directly). No partial index needed (the table is low-volume: one row
per token issuance, 5-minute TTL, purged by worker).

**Target part:** schema.ts Axioma section, migration 0004

### Finding 5 — HIGH — Full recommended DDL for schema.ts (Axioma section, after terminalLicenseEvents)

**Evidence:** Current Axioma section ends at `terminalLicenseEvents` (lines 567-579 in schema.ts).
The seed §DB schema groups lists `axioma_account_links`, `terminal_release_cache`,
`terminal_download_events`, `terminal_license_events` under Axioma — `axioma_handoff_jti_revocations`
is additive to this group.

**Recommended Drizzle DDL (append after `terminalLicenseEvents` export):**

```typescript
// --- Axioma: jti replay-prevention store (migration 0004) ---
// One row per issued handoff token. Atomic UPDATE used_at=now() WHERE jti=? AND used_at IS NULL
// AND revoked_at IS NULL AND expires_at>now() is the consume gate (0 rows = replay/expired/revoked).
// sub is NOT a FK (rows must survive user deletion for audit; app sweeps on delete).
// Bounded context: Axioma. Owner: db-architect. Source: docs/AXIOMA_HANDOFF_TOKEN_SPEC.md.
export const axiomaHandoffJtiRevocations = pgTable(
  'axioma_handoff_jti_revocations',
  {
    jti: uuid('jti').primaryKey(),                              // caller-supplied UUID v4 (no defaultRandom)
    sub: uuid('sub').notNull(),                                 // WTC user id — no FK; rows survive user delete
    issuedAt:  timestamp('issued_at',  { withTimezone: true }).notNull(),
    expiresAt: timestamp('expires_at', { withTimezone: true }).notNull(),
    usedAt:    timestamp('used_at',    { withTimezone: true }), // NULL = not yet consumed
    revokedAt: timestamp('revoked_at', { withTimezone: true }), // NULL = not revoked
    revokeReason: text('revoke_reason'),                        // set on manual/entitlement-revoke
  },
  (t) => ({
    expiresIdx: index('ahjr_expires_at_idx').on(t.expiresAt),   // worker purge: WHERE expires_at < ?
    subIdx:     index('ahjr_sub_idx').on(t.sub),                // per-user sweep: WHERE sub = ?
  }),
);
```

Note: `createdAt` is redundant with `issuedAt` for this table (spec uses `issued_at` as the
canonical creation time). No `createdAt()` helper needed.

**Target part:** schema.ts, migration 0004 DDL

### Finding 6 — HIGH — Repository function signatures (four functions recommended)

**Evidence:** Reference patterns in `repositories.ts`: `markExpiringSoon` (no per-row audit for
informational status bumps), `atomicRevokeTv` (system actor = `{id:null, role:'system'}`),
`insertWebhookEventOnce` (INSERT + onConflictDoNothing + returning), `sweepTvExpiry` (returns count).

**Recommended signatures (for implementation):**

```typescript
// 1. Record a newly-issued jti (called at token issuance, before signing)
export async function recordHandoffJti(
  db: Db,
  input: {
    jti: string;      // UUID v4 from crypto.randomUUID()
    sub: string;      // WTC user UUID
    issuedAt: Date;
    expiresAt: Date;
  },
): Promise<void>
// Implement: db.insert(s.axiomaHandoffJtiRevocations).values({...input}).
// No audit row at issuance (the caller — the API route — writes axioma.account_link_init; see Finding 8).

// 2. Atomic consume gate (called by /api/axioma/jti/consume route or inline on handoff issue)
export async function consumeHandoffJti(
  db: Db,
  jti: string,
  now: Date,
): Promise<{ consumed: boolean; reason?: 'already_used' | 'revoked' | 'expired' | 'not_found' }>
// Implement: attempt UPDATE used_at=now WHERE jti=? AND used_at IS NULL AND revoked_at IS NULL
//   AND expires_at > now RETURNING jti.
// 0 rows returned → check which sub-condition failed (SELECT to determine reason) → return consumed:false.
// 1 row returned → consumed:true.
// Note: the UPDATE is atomic; the SELECT for reason is best-effort diagnostic (race window acceptable).

// 3. Revoke one or all pending JTIs for a user (admin revoke or user-deletion sweep)
export async function revokeHandoffJtisByUser(
  db: Db,
  sub: string,
  revokeReason: string,
  now: Date,
): Promise<{ revoked: number }>
// Implement: UPDATE revoked_at=now, revoke_reason=? WHERE sub=? AND revoked_at IS NULL AND used_at IS NULL
//   RETURNING jti. Returns count. No audit row in this function (caller writes the audit).

// 4. Worker purge of expired rows (post-TTL cleanup, keeps a 1-hour buffer)
export async function purgeExpiredHandoffJtis(
  db: Db,
  now: Date,
  bufferMs: number,   // default: 60 * 60 * 1000 (1 hour)
): Promise<{ purged: number }>
// Implement: DELETE FROM axioma_handoff_jti_revocations WHERE expires_at < (now - buffer) RETURNING jti.
// Returns count. No audit row (bulk maintenance, not a user-visible access change).
```

### Finding 7 — HIGH — consumeHandoffJti: atomic UPDATE pattern is correct; no transaction needed for the consume gate

**Evidence:** `repositories.ts` `insertWebhookEventOnce` (line 1059-1090): uses a single
`INSERT ... ON CONFLICT DO NOTHING ... RETURNING` — one SQL round-trip, no explicit transaction.
The jti consume gate must be equally atomic. A single UPDATE with the full predicate in the WHERE
clause is the correct approach:

```sql
UPDATE axioma_handoff_jti_revocations
SET used_at = $now
WHERE jti = $jti
  AND used_at IS NULL
  AND revoked_at IS NULL
  AND expires_at > $now
RETURNING jti
```

Zero rows returned means exactly one of: jti not found, already used, revoked, or expired. The
predicate evaluation and the update are atomic under Postgres row-level locking. No `db.transaction()`
wrapper needed for the consume gate itself. If the caller writes an audit row alongside the consume
(e.g. `axioma.account_link.jti.consume`), that should go in a transaction.

**Target part:** repositories.ts `consumeHandoffJti`

### Finding 9 — MEDIUM — recordHandoffJti should NOT write an audit row; the issuing route does

**Evidence:** The spec §Token Issuance Flow step 8: "WTC writes audit log: `axioma.account_link.init`"
— this is written by the API route handler after signing the token, not by the DB insert function.
The `recordHandoffJti` function is a pure DB-layer insert (like `insertBotMetricSnapshot` — no audit).
This mirrors `markExpiringSoon` (no per-row audit for informational ops) and
`insertWebhookEventOnce` (pure insert, no audit).

Audit rows for jti lifecycle events should be written by the route/service layer, not the repo
function. This keeps the repo layer focused on data persistence and avoids coupling the jti insert
to an audit write that the caller may or may not want.

**Recommendation:** `recordHandoffJti` inserts only the jti row. The route that calls it also calls
`db.insert(s.auditLogs).values(...)` separately (or uses `createDbAuditWriter`). This mirrors the
current pattern throughout repositories.ts.

## Decisions

**D-1 — jti column type: `uuid` NOT `text`.**
The spec says `uuid PRIMARY KEY`. Use Drizzle `uuid('jti').primaryKey()` (caller-supplied, no
`defaultRandom()`). Postgres `uuid` gives format validation and more efficient indexing than `text`.

**D-2 — sub column: `uuid NOT NULL`, NO FK to users(id).**
Rows must survive user deletion for audit purposes. App-level sweep (revokeHandoffJtisByUser) handles
the user-delete revocation. Consistent with `audit_logs.actor_user_id` and
`billing_manual_review_items.user_id` precedents.

**D-3 — Two non-unique indexes: `ahjr_expires_at_idx` and `ahjr_sub_idx`.**
Supports the worker purge scan and the per-user revoke sweep respectively.

**D-4 — Migration 0004 scope is EXACTLY ONE TABLE: `axioma_handoff_jti_revocations`.**
No `axioma_account_links` changes. No other tables. Scope discipline enforced (B4/TARGET boundary).

**D-5 — Four repository functions: `recordHandoffJti`, `consumeHandoffJti`,
`revokeHandoffJtisByUser`, `purgeExpiredHandoffJtis`.**
`consumeHandoffJti` is a single atomic UPDATE (no transaction wrapper needed at the DB function
level). `recordHandoffJti` does NOT write an audit row (route layer writes it). Signatures as
specified in Finding 6.

**D-6 — Three new audit action codes to add to AUDIT_ACTIONS before implementing jti repos.**
Recommended names (underscore-consistent with existing axioma.* codes):
`axioma.handoff_jti_consume`, `axioma.handoff_jti_replay`, `axioma.handoff_jti_revoke`.
Must be added to `packages/audit/src/audit.ts` as part of the DB wave (audit.ts is on the
serialized spine).

**D-7 — `purgeExpiredHandoffJtis` added to worker `dbTick` after TV sweep.**
Uses a 1-hour buffer: `bufferMs = 60 * 60 * 1000`. Safe to run on every tick.

**D-8 — All jti store functions stay in `packages/db/src/repositories.ts`.**
`packages/axioma-bridge` is a zero-dependency pure package; it must not import `@wtc/db`.

## Risks

1. **Consume race (two concurrent consume calls for the same jti):** The single atomic UPDATE
   handles this correctly — exactly one wins (sets `used_at`); the other gets 0 rows and rejects
   as replay. No additional locking needed.

2. **Audit action name collision/inconsistency:** Existing Axioma codes use underscores
   (`axioma.account_link_init`), but the spec names the new codes with dots
   (`axioma.account_link.jti.consume`). The AUDIT_ACTIONS `as const` tuple enforces type-safety,
   but the inconsistency risks confusion. Resolved by D-6 (use underscore names).

3. **User-delete sweep timing:** If a user is deleted before `revokeHandoffJtisByUser` runs,
   a jti for that user could be consumed between deletion and sweep. This is acceptable: the
   5-minute TTL means the maximum exposure window is short. The spec explicitly notes this is
   "acceptable given the short TTL."

4. **PGlite integration test feasibility:** The consume path uses a Drizzle `update...returning()`
   call. PGlite supports this (Phase 2.x tests use returning() throughout repositories.ts). A
   cross-connection concurrent test (two PGlite connections race to consume the same jti) is
   feasible but PGlite's WAL mode serialization is not identical to Postgres locking semantics.
   Mark that specific cross-connection test as a "best-effort" confidence test, not a substitute
   for a real-PG concurrent test (B1 — real-PG still NOT RUN).

5. **No FK means no cascade safety net:** If a bug omits the `revokeHandoffJtisByUser` call on
   user delete, orphaned jti rows will persist until the purge worker cleans them. This is an
   acceptable risk (audit rows have the same property) and the TTL + purge worker bound the
   maximum outstanding row age to TTL + buffer (5 min + 1 hour = ~65 minutes at most for active
   tokens; expired tokens purge independently).
