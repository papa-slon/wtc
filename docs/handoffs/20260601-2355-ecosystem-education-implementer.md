## Findings
1. Severity: High. Evidence: `tests/e2e/lms-db-materials.spec.ts:164-216` closes the DB-backed LMS test at line 164 and then leaves a duplicated orphan code fragment at lines 165-216. This spec is not acceptable as current runnable evidence until that tail is removed. Recommendation: delete the duplicate tail after the first closing `});`, then run the opt-in command only through `npm run e2e:lms:db` with a fresh `wtc_test_lms_<timestamp>` database. Target part: DB-backed LMS browser spec.
2. Severity: Medium. Evidence: teacher UI supports creating lessons and materials through real server actions at `apps/web/src/app/teacher/courses/[id]/page.tsx:114-124` and `apps/web/src/app/teacher/courses/[id]/page.tsx:232-253`; actions sanitize embeds and prepare file uploads at `apps/web/src/features/lms/actions.ts:94-130` and `apps/web/src/features/lms/actions.ts:345-381`. The Playwright spec currently drives these forms by generic form/button locators at `tests/e2e/lms-db-materials.spec.ts:47-87`, which is brittle and can match the wrong form as the page grows. Recommendation: add stable test ids to the teacher course, lesson, and material forms and to created material rows, then update the spec to use those row-scoped locators. Target part: teacher LMS UI and DB e2e selectors.
3. Severity: Medium. Evidence: quarantined files are generated by real byte scanning in `packages/lms/src/materials.ts:117-156`; student material rows only expose a download URL when `scanStatus === 'clean'` at `apps/web/src/features/lms/queries.ts:64-81`; the student page renders `download unavailable` when no download URL exists at `apps/web/src/app/(app)/app/education/[courseId]/[lessonId]/page.tsx:141-157`; the DB download query also filters to clean, active, published files at `packages/db/src/repositories.ts:787-831`. Current browser assertions cover visible no-download UI at `tests/e2e/lms-db-materials.spec.ts:118-121`, but do not directly request the quarantined material route because the UI does not expose a stable material id. Recommendation: add a non-secret `data-material-id` and `data-testid` on LMS material rows, then have the DB spec request `/api/education/materials/<quarantined-id>/download` as the entitled user and assert fail-closed `404` or equivalent no-store denial plus no body bytes. Target part: quarantined-file no-download proof.
4. Severity: Medium. Evidence: sanitized embed storage is enforced by `packages/lms/src/materials.ts:207-242`; teacher actions store canonical sanitized HTML at `apps/web/src/features/lms/actions.ts:94-100` and `apps/web/src/features/lms/actions.ts:360-365`; student rendering reparses sanitized HTML and emits a React iframe with sandbox at `apps/web/src/app/(app)/app/education/[courseId]/[lessonId]/page.tsx:10-24` and `apps/web/src/app/(app)/app/education/[courseId]/[lessonId]/page.tsx:106-109,157`. Current spec asserts a material iframe title/src at `tests/e2e/lms-db-materials.spec.ts:122-123`, but the teacher-side iframe assertion at `tests/e2e/lms-db-materials.spec.ts:97-99` is weak and the lesson-level embed path is not created. Recommendation: add a lesson-level embed scenario and replace the weak iframe locator with row-scoped assertions for `src`, `title`, `sandbox`, `referrerpolicy`, absence of `srcdoc`, and absence of the raw submitted iframe string in page HTML. Target part: sanitized embed browser proof.
5. Severity: Medium. Evidence: clean download handling requires session, education entitlement, database, clean file row, and audit at `apps/web/src/features/lms/material-download.ts:47-66`; response headers include `private, no-store`, `content-disposition`, `nosniff`, and `x-lms-sha256` at `apps/web/src/features/lms/material-download.ts:31-40`; audit payload avoids raw bytes and storage key at `packages/db/src/repositories.ts:833-851`. Current spec checks success headers/body/hash at `tests/e2e/lms-db-materials.spec.ts:127-146`. Recommendation: keep this route-level `page.request.get()` proof, but also assert `content-length` equals the byte length and that admin audit pages do not render `fileBytesBase64`, `storageKey`, `lms/materials/`, the raw file text, or the file base64. Target part: clean download and no leak proof.
6. Severity: Medium. Evidence: student pages fail closed before data load when `accessFor(user.id, 'education')` denies access at `apps/web/src/app/(app)/app/education/[courseId]/[lessonId]/page.tsx:34-47`; student query functions also return null without access at `apps/web/src/features/lms/queries.ts:255-303`; the download route performs its own entitlement check at `apps/web/src/features/lms/material-download.ts:56-59`. Current seed creates only `admin@wtc.local`, `teacher@wtc.local`, and an entitled `user@wtc.local` at `packages/db/src/seed.ts:27-63`, so the browser spec cannot prove a denied student flow without using the teacher account as a proxy at `tests/e2e/lms-db-materials.spec.ts:132-136`. Recommendation: have the LMS DB e2e prep create an e2e-only `education-denied@wtc.local` user with no education entitlement in the throwaway database, add a `loginEducationDenied` helper, and assert `/app/education` locked UI plus direct download `403` no-store. Target part: student entitlement flow.
7. Severity: Low. Evidence: `apps/web/src/features/lms/actions.ts:54` still says embed is intentionally not selectable, while the same schema and UI now allow `contentType: 'embed'` at `apps/web/src/features/lms/actions.ts:60-65` and `apps/web/src/app/teacher/courses/[id]/page.tsx:119-123,197-201`. Recommendation: update the stale comment during the next implementation pass so source comments do not contradict the current product behavior. Target part: LMS maintainability.

## Decisions
- Treat Phase 3.19 as a read-only audit only; no product code or tests were changed.
- Treat the current DB browser spec as not accepted until the duplicate tail is removed and `npm run e2e:lms:db` is observed against a fresh throwaway Postgres database.
- Keep the next proof path browser-real and route-real: teacher UI creates data, student UI renders data, and `page.request` hits the real download route. Do not introduce fake fixtures or bypass route logic for acceptance.
- Keep local DB byte storage, deterministic scanner, and local e2e login as local acceptance tools only. They do not replace production object storage, production malware scanning, or public rollout approval.

## Risks
- The duplicated tail in `tests/e2e/lms-db-materials.spec.ts` means prior default e2e success does not prove the opt-in LMS DB spec is currently runnable.
- Adding `data-material-id` exposes internal UUIDs in HTML, but course and lesson IDs are already URL-visible; it is acceptable for route-level no-download proof if storage keys, raw bytes, and secrets remain hidden.
- Creating an e2e-only denied user in `prepare-lms-db-e2e.ts` must stay limited to the throwaway DB harness and must not alter general production seed semantics.
- Quarantined material route denial should not reveal scan reasons or storage metadata to students. UI can show safe status copy; route response should stay generic and no-store.
