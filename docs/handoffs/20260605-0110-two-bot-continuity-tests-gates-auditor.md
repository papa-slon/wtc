## Decisions
- Recommend a standalone `npm run accept:bots:continuity:contract` wrapper backed by a `bot-continuity-contract` gates.mjs mode.
- Recommend keeping the current `bot-admin-local` plan unchanged in the first implementation because it already runs `ci:local`, and root `npm test` will include the new test once added.
- Recommend the new focused test file name `tests/integration/two-bot-continuity-contract.test.ts`.
- Recommend no product-code edits for the gate unless the new test exposes a missing contract string or missing pure export.
## Risks
- If the contract gate is added to `bot-admin-local` as a separate child, it may duplicate part of `ci:local` because `ci:local` already runs `npm test`. That is acceptable if the operator wants explicit summary output, but it should be a deliberate choice.
- Static string tests can drift with harmless copy/refactor changes. Keep assertions on stable contract identifiers, tuple keys, safety boundaries, and exported functions rather than marketing copy.
- This cheap gate cannot prove managed worker continuity, real Postgres tuples, provider reachability, or live bot readiness. Those remain blocked by env/source/safety gates.
- The working tree was already heavily dirty before this audit, including package.json, scripts/gates.mjs, docs, and integration tests. Implementation should re-check current diffs before editing.
