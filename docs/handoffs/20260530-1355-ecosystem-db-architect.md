## Repository additions (packages/db/src/repositories.ts)

### Workstream D — Billing durable idempotency

**`insertWebhookEventOnce(db, input) → { isDuplicate, rowId }`**
- Uses `INSERT … onConflictDoNothing().returning()` on `(provider, event_id)` unique index
- 0 rows returned = duplicate. No select-then-insert race possible.
- Must be called OUTSIDE the entitlement-mutation transaction.

**`updateWebhookEventStatus(db, provider, eventId, status, productsChanged)`**
- Stamps final outcome on a `billing_webhook_events` row after processing.

**`createManualReviewItem(db, input) → ManualReviewItemRow | null`**
- `onConflictDoNothing` on `(provider, event_id)` — idempotent across retries.
- Writes audit row (`billing.manual_review_created`) in the same transaction.
- Returns `null` if item already exists.
- `eventSnapshot` MUST NOT include `NormalizedEvent.raw` or any PII/secrets.

**`listManualReviewItems(db, opts?) → ManualReviewItemRow[]`**
- Optional `status` filter + `limit`. Default limit 100. Returns most recent first.

**`resolveManualReviewItem(db, input)`**
- Updates review item + writes audit in ONE transaction.
- For `resolution==='approved'` with `approvalTarget`: calls `grantProduct()` SEPARATELY (own txn) after the update txn. This avoids PGlite nested-transaction/savepoint fragility (billing auditor R-01). Documented in code comment.
- Throws `manual_review_item_not_found` or `manual_review_item_already_resolved` on guard failures.

**`flagProductForReview(db, userId, productCode, now?, actorUserId?, reason?)`**
- Transitions any entitlement state → `manual_review` via `applyBillingEvent('flag_review', …)`.
- Writes audit (`product.flag_review`) + `product_access_events` row in same transaction.
- No-ops if no entitlement exists or already in `manual_review`.

**`upsertSubscription` (updated)**
- When `providerRef` is present: uses `INSERT … onConflictDoUpdate` on the new `subscriptions_user_provider_ref_idx` unique index. Safe under concurrency.
- When `providerRef` is absent: plain INSERT (one-time payments, manual grants).

### Workstream E — TradingView atomicity

**`revokeTv` (updated)**
- Added optional `reason?: string` trailing param. Existing callers compile unchanged.
- `reason` now threaded into the audit `after` payload.

**`atomicGrantTv(db, input, now?) → TvGrantRow`**
- ONE transaction: request status update → grant row insert → profile UPSERT → audit.
- Replaces the two-transaction `grantTv + createTvGrant` pattern (F-02: diverged state eliminated).
- `grantTv` and `createTvGrant` remain exported for `sweepTvExpiry` + existing tests.

**`atomicRevokeTv(db, requestId, adminId, reason?, now?)`**
- ONE transaction: request status stamp → find active grant by `requestId` (may not exist) → stamp grant `revokedAt/revokedBy/revokeReason` → null profile `currentGrantId` → audit.
- Graceful when no grant exists (F-03: `grantId` not needed as input).
- `reason` written to both grant row and audit payload.

### Workstream F — Admin N+1 fix

**`listUsersWithCreatedAt(db) → DbUserWithCreatedAt[]`**
- 2-query solution: one flat `SELECT` on `users`, one flat `SELECT` on `user_roles`, joined in-process (O(N)). Not N+1.
- Returns `createdAt` as epoch-ms.
- Type includes `passwordHash` — callers MUST strip before returning to clients (use `mapToAdminUserView`).

- **D-01:** `billing_webhook_events` table — richer shape per billing auditor (adds `userId`, `planCode`, `billingEvent`, `status`, `productsChanged`, `expiresAt`).
- **D-02:** NOT superseded — `billing_manual_review_items` IS implemented (billing auditor Decision 4 took precedence over db-architect D-02 which said no second table).
- **D-03 → D-06:** All implemented. See repository additions above.
- **D-07:** Test matrix implemented as `tests/integration/db-0003.test.ts` (14 tests, all green).
- **subscriptions unique index:** Plain `uniqueIndex` on `(user_id, provider, provider_ref)` — no partial WHERE. Postgres treats NULLs as distinct in a plain uniqueIndex so multiple NULL `provider_ref` rows are permitted. Simpler than a partial index.
- **PGlite savepoint avoidance:** `resolveManualReviewItem` does NOT call `grantProduct` inside a transaction. Grant happens SEPARATELY. Documented in code comment per billing auditor R-01.
