## Decisions
- Recommended next implementation shape: a no-env local/static fixture such as `tests/integration/two-bot-continuity-contract-static.test.ts`, optionally wrapped by a small `accept:worker:continuity:fixture` command, that imports pure helpers or scans source text only and never requires provider URLs or DB URLs.
- The fixture should prove three classes of assertions: provenance labels (`fixture/no-env`, `tortila-mock`, no `legacy-db`), blocked/skipped states (`legacy` not configured/blocked, live controls disabled), and analytics boundaries (Legacy closed-trade performance pending; Tortila/Legacy warnings persistent).
- The fixture must not replace `npm run accept:worker:continuity:managed`, managed DB matrix E2E, source-proof work, deploy proof, or production monitoring.
- No Legacy importer work should start until a separate source-proof artifact names the real table/API, provider filter, stable trade/fill id, economics, timestamps, raw allowlist, and replay semantics.
- No background agents were spawned by this single auditor lane; none needed cleanup.

## Risks
- `mode: 'real'` on the Legacy blocked adapter is easy to misread unless the fixture asserts B3/not-configured/down/no-network semantics.
- `sourceAdapter: 'tortila-mock'` can make a local dashboard nonblank; without provenance assertions, screenshots or continuity tuples can be overstated as live journal proof.
- Legacy active slots/orders/stage coverage are useful operational evidence but can tempt false win-rate/PF/realized-PnL claims.
- A warning-present continuity result can look "green" if tests only check `ok`; warnings must remain part of acceptance semantics.
- `worker-smoke` and no-env fixture proof are useful local guardrails, but they do not prove the managed tuple `worker_status=ok/bot_continuity=ok/tortila=ok/legacy=ok`.
