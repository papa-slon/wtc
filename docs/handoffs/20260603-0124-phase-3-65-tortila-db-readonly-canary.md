# Phase 3.65 Tortila DB-backed read-only canary handoff
## Scope
Phase 3.65 moved Tortila from mock-only canary visibility to a real read-only production canary path:
Tortila journal -> WTC worker -> WTC Postgres snapshots/imports -> WTC web/admin UI.

This phase did not enable live bot control, did not enable Legacy real integration, did not enable Axioma live routes, did
not run Stripe billing acceptance, and did not modify the live bot code or bot configuration. Tortila real reads are accepted
only for the WTC DB-backed read-only canary. Live start/stop/apply-config remains disabled.

## Findings
1. Severity: High. The original Phase 3.64 canary was mock-only for bots. The first Phase 3.65 read-only audits correctly
   rejected a shared web `BOT_ADAPTER_MODE=read-only` path because it could let user page renders direct-fetch the Tortila
   journal. Recommendation: accept only the later DB-backed implementation where production Tortila UI reads WTC Postgres
   snapshots/imports, not the live journal directly. Target part: web data path.
2. Severity: High. The worker now writes real Tortila read-only evidence into WTC DB. Evidence from the managed worker loop:
   `tortila-journal` and `worker` health rows are `ok`; metric snapshots exist; position snapshots exist; trade imports are
   idempotent and show `13` imported trades. Recommendation: keep the worker container monitored and treat append-only
   metric/position rows as time-series evidence. Target part: worker canary.
3. Severity: High. Live bot services remained active and were not changed. Evidence: `turtle-bot.service`,
   `turtle-journal.service`, `wtc-bot-api-firewall.service`, `nginx`, and `postgresql` were active after WTC deploy; only
   WTC canary/worker containers were changed. Recommendation: keep future WTC work scoped to WTC services unless a separate
   bot-owner maintenance phase is explicitly approved. Target part: bot service safety.
4. Severity: High. Bot API public exposure is still controlled by firewall rather than loopback binding. Evidence: external
   probes showed `80` and `443` open while `8000` and `8080` timed out/closed; server-local bot ports remained reachable.
   Recommendation: keep `wtc-bot-api-firewall.service` active and monitored; prefer loopback/private binding in a separate
   bot maintenance phase. Target part: network boundary.
5. Severity: High. Provider-side journal bearer auth remains not proven. Evidence: WTC uses a read token in its environment,
   but earlier no-token journal health probing showed provider-side unauthenticated health was reachable. Recommendation:
   do not call the journal auth gate green until no-token requests to mapped data endpoints fail, valid-token requests pass,
   and retained logs/artifacts prove no token leakage. Target part: journal auth.
6. Severity: High. Live control remains disabled. Evidence: `FEATURE_LIVE_BOT_CONTROL=false`, WTC Tortila HTTP control
   methods are disabled, and browser/admin checks showed no start/stop/apply-config controls. Recommendation: keep control
   disabled until separate security and bot-integration control gates pass. Target part: live control.
7. Severity: Medium. Legacy remains blocked. Evidence: Legacy non-mock path is still a blocked adapter due plaintext
   exchange-key risk. Recommendation: do not probe or enable Legacy authenticated API in this canary. Target part: Legacy.
8. Severity: Medium. This phase is a real Tortila read-only canary, not full production. Stripe self-serve billing, Axioma
   live bridge/download/account-link acceptance, live LMS object-store/scanner, branded-domain DNS/TLS, long burn-in, and
   full production monitoring remain separate gates. Target part: production truth.

## Decisions
- Accept commit `4487b3d` as the current Tortila DB-backed read-only canary implementation.
- Accept server release `20260602-1816-4487b3d` as the current WTC canary release.
- Run `wtc-ecosystem-canary` for web and `wtc-ecosystem-worker` for the managed worker loop.
- Keep `wtc-ecosystem-preview` available as rollback.
- Keep `FEATURE_LIVE_BOT_CONTROL=false`, `FEATURE_TV_AUTOMATION=false`, and Axioma routes disabled.
- Keep Legacy blocked/non-live.
- Do not document raw public host coordinates, IPs, DB URLs, tokens, cookies, or bot secrets in durable docs.

## Risks
- Provider-side journal bearer auth is still not accepted green; current safety relies on WTC token usage plus server network
  isolation.
- Firewall drift could expose bot API listeners because the underlying services still listen broadly.
- Worker metric/position snapshots are append-only; repeated worker ticks create additional rows by design.
- Real UI freshness depends on the managed Docker worker continuing to run and write snapshots.
- Full production remains incomplete until provider, billing, LMS, Axioma, branded-domain, monitoring, and live-control gates
  are separately cleared.
