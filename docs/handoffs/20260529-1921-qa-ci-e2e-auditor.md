## Findings

### Finding 1 — HIGH (H1/C): analytics has no Vitest test file; coverage runs via smoke only

- **Severity:** high
- **Evidence:** `packages/analytics/src/__smoke__.ts` exists and is exercised by `npm run check:core`. Glob `packages/analytics/**/*.test.ts` → no files found. `vitest.config.ts:7` includes only `packages/**/*.test.ts` and `tests/integration/**/*.test.ts`. The analytics module is therefore absent from `npm test` and `npm run coverage` output.
- **Impact:** `computeMetrics`, `computeDrawdown`, closed-vs-unrealized PnL distinction, and null-not-zero drawdown paths are not in the Vitest coverage report. The 23.75% statement coverage figure (`docs/STATUS.md:19`) under-counts coverage AND the analytics module is not enforced by the Vitest test matrix.
- **Recommendation:** Create `packages/analytics/src/metrics.test.ts` as a proper Vitest test that mirrors the 14 assertions already in `__smoke__.ts`. This brings analytics normalization (closed vs unrealized PnL, drawdown, null-not-zero) into `npm test` and `npm run coverage` where it belongs. Smoke can remain for zero-install pre-install sanity checks.
- **Target part:** C / H

### Finding 2 — HIGH (H2): No .github/workflows directory — CI does not exist

- **Severity:** high
- **Evidence:** Glob `.github/workflows/**` → no files found (confirmed). No `.github` directory. `docs/STATUS.md:47` confirms: "add CI workflow" is still on the TODO list. `docs/handoffs/20260529-phase1-persistence-hardening.md:76` states: "add CI workflow" under "Still NOT deployable until".
- **Impact:** Every gate listed in STATUS.md (lint, typecheck, test, build, e2e, secret scan, coverage) is operator-run manually. There is no automated enforcement on push/PR. Secrets are not scanned automatically. Coverage regression goes undetected.
- **Recommendation — draft GitHub Actions workflow:** Create `.github/workflows/ci.yml` with:
  - Trigger: `push` + `pull_request` to `main`.
  - `services: postgres:16` container at 5432 (`POSTGRES_PASSWORD: wtc`, `POSTGRES_DB: wtc`) with `pg_isready` health check.
  - Steps in order:
    1. `actions/checkout@v4`
    2. `actions/setup-node@v4` with node 24 + npm cache
    3. `npm ci`
    4. `npm run check:core`
    5. `npm run lint`
    6. `npm run typecheck` and `npm run typecheck -w @wtc/web` (parallel)
    7. `npm run secret:scan`
    8. `npm test` (Vitest, includes PGlite integration)
    9. `npm run coverage` (optionally upload artifact)
    10. `npm run build -w @wtc/web`
    11. `npx playwright install --with-deps chromium`
    12. `npm run e2e` (with `CI=true` so `reuseExistingServer` is false per `playwright.config.ts:26`)
    13. (Phase 1.5+) `DATABASE_URL=postgres://wtc:wtc@localhost:5432/wtc npm run db:migrate -w @wtc/db && npm run db:seed -w @wtc/db` — real Postgres integration path
  - `env: CI: true` at workflow level.
  - **Local CI-equivalent script:** add a root script `"ci:local"` that chains: `npm run check:core && npm run lint && npm run typecheck && npm run typecheck -w @wtc/web && npm run secret:scan && npm test && npm run coverage && npm run build -w @wtc/web && npx playwright install chromium && npm run e2e`
- **Target part:** H

### Finding 3 — MEDIUM (H1): `npm run e2e` screenshot path is relative — fails silently if `tests/e2e/screenshots` does not exist on a fresh clone

- **Severity:** medium
- **Evidence:** `tests/e2e/smoke.spec.ts:4`: `const shot = (name: string, project: string) => \`tests/e2e/screenshots/\${name}-\${project}.png\`` — relative path resolved from CWD. `playwright.config.ts` has `screenshot: 'only-on-failure'` (line 13) not `'on'`, but the spec calls `page.screenshot({path: ...})` unconditionally on every passing test. The `tests/e2e/screenshots/` directory exists now (16 PNGs found via Glob) but is not committed (no .git repo) so a fresh clone would miss it.
- **Recommendation:** Add a `tests/e2e/screenshots/.gitkeep` file to the repo so the directory exists on fresh clones. Or add a `playwright.config.ts` global setup that creates the directory. In the GitHub Actions workflow, upload `tests/e2e/screenshots/**` as a job artifact after the e2e step.
- **Target part:** H

### Finding 4 — MEDIUM (H3): Playwright `--with-deps chromium` needed in CI; README/docs only document the `install chromium` form

- **Severity:** medium
- **Evidence:** `docs/STATUS.md:16`, `docs/NEXT_ACTIONS.md:21`, `docs/DEPLOYMENT.md:29` all document the prerequisite as `npx playwright install chromium`. On a Linux CI runner (Ubuntu), `playwright install chromium` alone is insufficient — OS-level browser dependencies also need `--with-deps`. The current docs and the suggested CI command use the shorter form.
- **Recommendation:** In the GitHub Actions workflow use `npx playwright install --with-deps chromium`. Document the distinction: on Windows developer machines `install chromium` suffices; on bare Linux CI runners `--with-deps` is required. Alternatively use `@playwright/test`'s dedicated GitHub Action (`playwright/msft-playwright-github-action` or the built-in `--with-deps` flag).
- **Target part:** H

### Finding 5 — MEDIUM (H4): PGlite integration test covers the ORM layer but a real-Postgres test would additionally need: migration DDL idempotency, constraint names, trigger/role enforcement, and connection pooling

- **Severity:** medium
- **Evidence:** `tests/integration/db-persistence.test.ts:1-7`: the test applies the real generated migration SQL (`packages/db/migrations/0000_*.sql`) to an in-process PGlite WASM engine. This verifies the Drizzle ORM query layer and repository business logic thoroughly (7 tests: seed, user round-trip, duplicate-email, grant/revoke, sealed keys, audit redaction, TV sweep).
- **What a real-Postgres CI test would additionally need:**
  1. Run `db:migrate` via `drizzle-kit migrate` (tests the CLI and PG migration tracking table).
  2. Run `db:seed` via `tsx seed-cli.ts` (tests the CLI entry-point, not just the library function).
  3. Test `DATABASE_URL` connection pooling with `postgres-js` (the production driver — PGlite uses a different adapter path).
  4. Verify `audit_logs` append-only enforcement if a restricted DB role is later created.
  5. Verify `exchange_api_key_secrets` column-level permissions if column encryption is added.
  6. Test concurrent `createUser` duplicate-email rejection under real transaction isolation (PGlite is single-threaded).
  7. Test `ON DELETE CASCADE` / FK constraint behavior (PGlite does enforce FK but subtle constraint names differ).
- **Gate commands (runnable status):**
  - `npm test` (Vitest + PGlite): RUNNABLE here (no Docker needed).
  - `npm run db:migrate -w @wtc/db` + `npm run db:seed -w @wtc/db` against real Postgres: NOT RUNNABLE on this host (no Docker; `DATABASE_URL` unknown for local PostgreSQL 17 at 127.0.0.1:5432 — credentials unknown per host facts).
  - Real-Postgres integration test in CI (GitHub Actions postgres:16 service): NOT RUN (no CI workflow exists yet).
- **Target part:** H

### Finding 8 — LOW (H3): E2e runs `next dev` (not `next build + start`) — production code paths not exercised

- **Severity:** low
- **Evidence:** `playwright.config.ts:22`: `command: 'npm run dev:e2e -w @wtc/web'` — this runs `next dev --port 3100`. `apps/web/package.json:7`: `"dev:e2e": "next dev --port 3100"`. The e2e suite therefore tests the development code path, not the production build. The `next.config.js` has `ignoreDuringBuilds: true` for ESLint, so the build path also differs.
- **Impact:** E2e passing does not guarantee the production build serves the same behavior. For example, React Server Component hydration differences, build-time static generation failures, or edge runtime incompatibilities would not be caught.
- **Recommendation:** Add an optional `e2e:prod` script that runs `npm run build -w @wtc/web && npx cross-env PORT=3100 npm run start -w @wtc/web` as the webServer command, and run it in CI after the regular e2e. Keep the dev-mode e2e for fast local iteration.
- **Target part:** H

## Decisions

1. Analytics smoke (`__smoke__.ts`) is valid for zero-install pre-install sanity but is NOT a substitute for a Vitest test in coverage. Both should exist.
2. The PGlite integration test is the right "no Docker needed" gate for the ORM/repository layer. A real-Postgres CI job is additive, not a replacement.
3. No `.github/workflows` directory exists — CI must be created from scratch.
4. All task-listed gate npm scripts are present; no script is missing.
5. The e2e dedicated port (3100) via `dev:e2e` + `playwright.config.ts:22` is the correct isolation approach. The `reuseExistingServer: !process.env.CI` pattern (line 26) is correct for CI.

## Risks

1. **No CI**: any regression in lint, typecheck, tests, secret scan, or e2e is only caught by manual operator execution. A broken commit can sit undetected.
2. **Analytics coverage gap**: the analytics normalization logic (closed vs unrealized PnL, drawdown, null-not-zero) is validated only via the smoke test which runs outside Vitest — coverage tooling does not see it.
3. **Real Postgres not tested**: `db:migrate` + `db:seed` have never run against a real Postgres instance (only PGlite). DDL edge cases (constraint names, FK cascades under concurrent load, postgres-js connection pooling) are unverified.
4. **E2e screenshots directory**: absent on fresh clones if the directory is not committed. All 16 screenshot PNGs are present currently but may be lost on a re-clone without a `.gitkeep`.
5. **HS256 handoff token**: `packages/axioma-bridge/src/handoff.test.ts:27` confirms the handoff is HS256 dev stub. This is tested and documented but remains a production blocker.
6. **Coverage thresholds not enforced**: regression in security-core coverage is silent.
