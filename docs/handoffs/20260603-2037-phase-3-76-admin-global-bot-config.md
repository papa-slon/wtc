## Findings
1. Severity: High. Existing user bot config storage is instance-owned and cannot truthfully model admin system defaults. Evidence: `bot_configs` is keyed by `bot_instance_id`; new tables are keyed by `product_code/profile_code`. Recommendation: keep global defaults separate from user overrides and runtime snapshots. Target part: DB model and admin UX.
2. Severity: High. Admin system defaults need durable audit/history and must not work in memory-only demo mode. Evidence: `/admin/bots/config` disables saving without Postgres and `adminSaveBotGlobalConfigAction` throws if `getServerDb()` is null. Recommendation: continue treating built-in defaults as review-only fallback until DB is configured. Target part: admin persistence.
3. Severity: High. Global defaults must not leak provider identities or secrets. Evidence: admin action rejects forbidden keys such as `apiKey`, `secret`, `token`, `providerPubId`, `providerAccountId`, `rawJson`, provider URLs, and live-control fields before DB writes. Recommendation: keep Legacy provider mapping in user/admin mapping surfaces only. Target part: schemas/actions/tests.
4. Severity: Medium. Browser plugin auth could not complete because this Browser backend disallowed clipboard-backed text input and cookie mutation from read-only evaluate. Recommendation: use the Playwright e2e runner for authenticated visual checks until the Browser surface supports safe login/cookie setup. Target part: browser acceptance workflow.

## Decisions
1. Route is `/admin/bots/config`; nav label is `Bot Defaults`.
2. Current system profile key is `system_default`; each product has one published, inherited default via a partial unique index.
3. `bot_global_configs` stores current rows; `bot_global_config_versions` stores immutable version history.
4. Audit action is `bot.global_config.save`, with metadata-only before/after payloads. Raw config JSON is stored in version history, not in audit rows.
5. Admin action follows `requireUser -> assertAdmin -> assertCsrf -> Zod -> product config validation -> forbidden-key scan -> DB repo -> revalidate`.
6. Page copy and tests preserve the boundary: WTC reference only, user settings unaffected, live control disabled.

## Risks
1. User-facing settings still do not consume the new system-default table as an inherited source. Current user settings continue to use built-in defaults or user-owned saved versions. Next phase should add resolved source labels and "Use system default / Customize my settings".
2. `npm run evidence:visual` remains fail-closed for pre-existing retained screenshots without a review manifest. The new screenshot generated by the e2e run was removed so this phase does not add retained visual artifact debt.
3. Full-suite `npm test` and full `npm run e2e` were not run in this phase; focused DB/static/admin e2e gates were run instead.
